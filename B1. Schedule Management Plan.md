Dokumen ini mengatur bagaimana jadwal proyek KANMIS 2.2 akan dikelola - mulai dari perencanaan, pelaksanaan, sampai monitoring. Target waktu kita adalah **150 hari kerja atau sekitar 5 bulan efektif** dengan nilai kontrak **Rp 919 juta** (lihat KAK BSN halaman 3).

## 2. PENDEKATAN MANAJEMEN JADWAL

### 2.1 Metodologi yang Digunakan

**Metode Utama: Critical Path Method (CPM)**

Schwalbe (2019, hal 238) menjelaskan bahwa CPM adalah teknik network diagramming untuk memprediksi total durasi proyek. Ini tools penting untuk mencegah project overrun.

Alasan kita pilih CPM:

- Deadline proyek sudah fix di 150 hari (requirement dari KAK)
- Task dependencies cukup clear karena pakai pendekatan waterfall
- Resource yang terbatas perlu dioptimalkan
- Stakeholder butuh prediksi yang akurat untuk payment milestones

**Teknik Pendukung:**

1. Precedence Diagramming Method (PDM) untuk visualisasi network
2. Forward & Backward Pass untuk hitung Early Start/Finish dan Late Start/Finish
3. Float Analysis untuk identifikasi critical vs non-critical activities
4. Resource Leveling untuk optimasi SDM
5. What-If Analysis untuk planning skenario risk

### 2.2 Proses Penjadwalan

Mengikuti framework PMBOK, kita menjalankan proses:

1. **Plan Schedule Management** → Dokumen yang sedang Anda baca ini
2. **Define Activities** → Breakdown jadi 89 work packages dari WBS
3. **Sequence Activities** → Bikin network diagram pakai PDM
4. **Estimate Activity Durations** → Pakai expert judgment + data historis
5. **Develop Schedule** → Analisis CPM + buat Gantt Chart
6. **Control Schedule** → Tracking mingguan dengan Earned Value Management

## 5. BASIS ESTIMASI DURASI

### 5.1 Metode Estimasi yang Dipakai

**Expert Judgment**: Kita manfaatkan pengalaman tim:

- PM dengan 6 tahun pengalaman
- SA dengan 4 tahun pengalaman
- Senior Programmers rata-rata 5 tahun

**Analogous Estimation**: Kita pakai data dari proyek sejenis:

- Project reference: SIMPONI Integration di Kemenkeu butuh 120 hari
- Accreditation system upgrades biasanya 4-6 bulan
- Adjustment factor: +25% untuk kompleksitas KANMIS (40+ fitur)

**Three-Point Estimates** untuk aktivitas high-risk: Pakai rumus (Optimistic + 4×Most Likely + Pessimistic) / 6

Contoh untuk Integrasi SIMPONI:

- Optimistic: 5 hari
- Most Likely: 7 hari
- Pessimistic: 12 hari
- Expected Duration = (5 + 28 + 12) / 6 = 7.5 hari

### 5.2 Asumsi Ketersediaan Resource

Berdasarkan Kualifikasi SDM di KAK (halaman 12-13):

- PM: Full-time 6 bulan
- SA: Full-time 6 bulan
- Senior Programmers (3 orang): Kombinasi full-time
- DBA: 4 bulan
- UI/UX: 3 bulan
- QA/Tester: 2 bulan
- Security Tester: 0.5 bulan

## 6. PROSEDUR KONTROL JADWAL

### 6.1 Monitoring Progress

**Weekly Progress Meeting**:

- **Jadwal**: Setiap Jumat jam 14:00-16:00 WIB
- **Peserta**: PM, SA, Lead Developers, PPK (kalau perlu)
- **Output**: Weekly status report

**Daily Standup** (fase Development):

- **Durasi**: 15 menit
- **Format**: Kemarin ngapain, hari ini mau ngapain, ada blocker apa
- **Tim**: PM + Development team

### 6.2 Pengukuran Kinerja Jadwal

**Metrik Earned Value Management (EVM)**:

|Metrik|Formula|Interpretasi|
|---|---|---|
|Schedule Variance (SV)|EV - PV|Negatif = terlambat|
|Schedule Performance Index (SPI)|EV / PV|< 1.0 = terlambat, > 1.0 = lebih cepat|
|Estimated Time to Complete|Durasi Original / SPI|Proyeksi waktu selesai|

Schwalbe (2019, hal 284) menjelaskan: "Schedule variance adalah earned value dikurangi planned value. SV negatif artinya pekerjaan butuh waktu lebih lama dari rencana."

**Threshold Kinerja**:

- **Hijau**: SPI ≥ 0.95 → on track
- **Kuning**: 0.85 ≤ SPI < 0.95 → perlu perhatian
- **Merah**: SPI < 0.85 → corrective action wajib

### 6.3 Change Control untuk Jadwal

**Proses Change Request**:

1. PM dokumentasikan impact analysis
2. Evaluasi dampak ke critical path
3. Kalau impact >5 hari → butuh approval CCB
4. Kalau disetujui → update schedule baseline
5. Notifikasi ke semua stakeholder

**Perubahan Baseline Jadwal** butuh:

- Approval tertulis dari PPK (Habib Nurhasan)
- Dokumentasi: Change log, revised Gantt chart, impact analysis
- Komunikasi via email + meeting formal dengan steering committee

## 7. RISK MANAGEMENT UNTUK JADWAL

### 7.1 Top 5 Schedule Risks

|Risk|Probabilitas|Impact|Mitigasi|
|---|---|---|---|
|Integrasi SIMPONI delay|Tinggi|Tinggi|Engagement awal ke Kemenkeu, buffer time|
|Resource turnover|Sedang|Tinggi|Knowledge transfer, backup resources|
|Scope creep|Sedang|Tinggi|Strict change control|
|UAT approval delay|Sedang|Sedang|Regular preview ke stakeholder|
|Kompleksitas teknis|Tinggi|Sedang|Prototype approach|

### 7.2 Schedule Buffers

**Project Buffer**: 15 hari (10% dari durasi)

- Tidak ditunjukkan di jadwal client-facing
- Hanya dipakai dengan approval CCB
- Untuk protect final deadline

**Feeding Buffers**:

- Integrasi SIMPONI: +5 hari
- Fase UAT: +3 hari

## 11. KESIMPULAN & REKOMENDASI

### 11.1 Faktor Sukses Utama

1. **Strict Critical Path Management**: Zero tolerance untuk delay di critical activities
2. **Early SIMPONI Engagement**: Koordinasi dengan Kemenkeu dari minggu pertama
3. **Resource Stability**: Minimize turnover, terutama Senior Programmers
4. **Stakeholder Engagement**: Regular UAT preview untuk hindari last-minute changes

### 11.2 Rekomendasi

1. Setup automated progress tracking tools di minggu pertama
2. Review CPM setiap 2 minggu
3. Komunikasi erat dengan PPK untuk milestone approvals
4. Dokumentasi lessons learned setiap fase untuk continuous improvement
