## RINGKASAN EKSEKUTIF

Dokumen ini menyajikan spesifikasi teknis komprehensif untuk pengembangan aplikasi KANMIS (KAN Management Information System) Versi 2.2 berbasis web bagi Badan Standardisasi Nasional (BSN). Solusi yang diusulkan dirancang untuk memenuhi seluruh persyaratan teknis yang tertuang dalam **Kerangka Acuan Kerja (KAK) Halaman 7-8, Tabel 2**, dengan menghadirkan sistem manajemen akreditasi yang scalable, aman, dan berkinerja tinggi untuk melayani lebih dari **2.700 Lembaga Penilaian Kesesuaian (LPK)** di seluruh Indonesia.

Pendekatan teknis kami memanfaatkan teknologi enterprise-grade yang telah terbukti, dengan integrasi penuh ke sistem pembayaran SIMPONI (Sistem Informasi PNBP Online), memastikan proses pembayaran yang seamless dan kepatuhan terhadap regulasi keuangan pemerintah.

**Keunggulan Teknis Utama:**

- ✅ Kepatuhan penuh terhadap spesifikasi teknis KAK
- ✅ Integrasi sistem pembayaran SIMPONI
- ✅ Keamanan enterprise (CAPTCHA, QR Digital Signature, SSL/TLS)
- ✅ Arsitektur three-tier client-server
- ✅ Technology stack terbukti (Laravel, PHP, SQL Server 2016)

## 1. ARSITEKTUR SISTEM

### 1.1 Arsitektur Web Three-Tier

**Rujukan:** KAK Bagian 1.e, Tabel 2, Halaman 7 - "Arsitektur: Client Server Data"

Sistem KANMIS 2.2 dibangun dengan arsitektur three-tier client-server yang robust sesuai spesifikasi KAK. Arsitektur ini memberikan pemisahan yang jelas antara presentation, business logic, dan data layer, memungkinkan skalabilitas, maintainability, dan keamanan optimal:

#### **TIER 1: PRESENTATION LAYER (Lapisan Klien)**

- **Antarmuka Web Responsif:** Dapat diakses melalui browser modern (Chrome, Firefox, Edge, Safari)
- **Dukungan Mobile:** Desain responsif untuk smartphone dan tablet
- **UI Framework:** Bootstrap (sesuai KAK) untuk konsistensi pengalaman pengguna
- **Progressive Web App (PWA):** Kemampuan offline untuk keandalan sistem
- **Teknologi AJAX:** Interaksi pengguna yang seamless tanpa reload halaman

#### **TIER 2: APPLICATION LAYER (Lapisan Logika Bisnis)**

- **Framework:** Laravel ver.5.7 sampai 10.41.0 (**Sesuai KAK**)
- **Runtime:** PHP 7.1.3 sampai 8.1.1 (**Sesuai KAK**)
- **Arsitektur API:** RESTful API untuk integrasi sistem
- **Business Rules Engine:** Pengelolaan alur kerja akreditasi
- **Session Management:** Penanganan state pengguna yang aman
- **Integration Middleware:** Konektivitas SIMPONI, SISPK, BSrE

#### **TIER 3: DATA LAYER (Lapisan Data)**

- **DBMS:** Microsoft SQL Server 2016 (**Sesuai KAK**)
- **High Availability:** Database clustering dan replikasi
- **Strategi Backup:** Backup otomatis dan recovery
- **Keamanan:** Enkripsi data at rest dan in transit
- **ACID Compliance:** Jaminan integritas transaksi

#### **Rasionalisasi Pemilihan Arsitektur**

Berdasarkan **Schwalbe Chapter 8 - Project Quality Management**, arsitektur three-tier memenuhi standar kualitas:

> "Quality is the degree to which the project fulfills requirements" (Schwalbe, p.318)

**Manfaat yang selaras dengan tujuan proyek:**

1. **Skalabilitas:** Setiap tier dapat diskalakan secara independen - krusial untuk 2.700+ pengguna LPK
2. **Keamanan:** Pendekatan defense-in-depth (KAK Lampiran II - Persyaratan Keamanan)
3. **Maintainability:** Perubahan terisolasi per tier - mengurangi risiko deployment
4. **Performa:** Distribusi beban mengoptimalkan utilisasi resource
5. **Fleksibilitas:** Upgrade teknologi per komponen - desain future-proof

### 3.1 Integrasi SIMPONI (Persyaratan Kritis)

**Rujukan:** KAK Halaman 7 - "Tujuan III: Kemampuan Integrasi dengan SIMPONI"

Integrasi SIMPONI (Sistem Informasi PNBP Online) merupakan **fitur breakthrough** KANMIS 2.2 untuk pemrosesan pembayaran otomatis.

#### **Metode Integrasi: REST API**

**Rujukan:** KAK Tabel 2 - "REST API Integration SIMPONI"

```
KANMIS 2.2                  SIMPONI (Kemenkeu)
    │                             │
    │ 1. Permohonan Akreditasi    │
    │────────────────────────────>│
    │                             │
    │ 2. Create Kode Billing (API)│
    │<────────────────────────────│
    │                             │
    │ 3. Tampilkan Kode Billing   │
    │   kepada LPK                │
    │                             │
    │ 4. Notifikasi Pembayaran    │
    │<────────────────────────────│
    │                             │
    │ 5. Update Status Pembayaran │
    │   & Lanjutkan Workflow      │
```

#### **API Endpoints**

**Outbound (KANMIS → SIMPONI):**

1. `POST /api/simponi/create-billing` - Generate kode billing
2. `GET /api/simponi/payment-status/{billing_code}` - Cek status pembayaran

**Inbound (SIMPONI → KANMIS):**

1. `POST /webhook/simponi/payment-notification` - Konfirmasi pembayaran callback
2. `POST /webhook/simponi/payment-cancellation` - Pemberitahuan pembatalan pembayaran

#### **Format Pertukaran Data**

```json
{
  "billing_request": {
    "lpk_id": "LPK-12345",
    "jenis_layanan": "akreditasi_baru",
    "jumlah": 5000000,
    "deskripsi": "Biaya Akreditasi Laboratorium",
    "tanggal_kadaluarsa": "2024-12-31"
  },
  "billing_response": {
    "kode_billing": "1234567890123456",
    "jumlah": 5000000,
    "tanggal_kadaluarsa": "2024-12-31",
    "url_pembayaran": "https://simponi.kemenkeu.go.id/pay/..."
  }
}
```

#### **Manfaat (Keselarasan KAK)**

Sesuai **Schwalbe Chapter 3 - Project Integration Management**:

> "Integration management involves making trade-offs among competing objectives" (Schwalbe, p.121)

**Analisis Trade-off:**

- ✅ **Eliminasi Input Data Manual:** Mengurangi human error dari input dual-system
- ✅ **Validasi Pembayaran Real-time:** Kelanjutan workflow instant
- ✅ **Transparansi:** LPK melacak status pembayaran online
- ✅ **Kepatuhan:** Adherensi regulasi PNBP Kemenkeu
- ✅ **Audit Trail:** Riwayat pembayaran lengkap untuk pelaporan

#### **Kebijakan Password (WAJIB)**

**Rujukan:** KAK Lampiran II, Point 4 - "Kendali Keamanan Kata Sandi"

|Persyaratan|Implementasi|Rujukan KAK|
|---|---|---|
|**Panjang Minimum**|8 karakter|Point 4a|
|**Case Sensitivity**|Huruf besar & kecil wajib|Point 4b|
|**Karakter Numerik**|Minimal 1 digit (0-9)|Point 4c|

```php
    'required',
    'string',
    'min:8',
    'regex:/[a-z]/',      // huruf kecil
    'regex:/[A-Z]/',      // huruf besar
    'regex:/[0-9]/',      // digit
    'confirmed'
]
```

### 4.4 Audit Logging

**Rujukan:** KAK Lampiran II, Point 5 - "Pencatatan Akses dan Aktivitas"

**Retensi Log Wajib:** Minimum 3 bulan

**Event yang Dicatat:**

- ✅ Login sukses/gagal dengan timestamp & IP
- ✅ Event logout
- ✅ Percobaan ubah password
- ✅ Modifikasi data (operasi CRUD)
- ✅ Upload/download file
- ✅ Perubahan permission
- ✅ Panggilan API ke sistem eksternal

**Penyimpanan Log:**

```sql
CREATE TABLE audit_logs (
    id BIGINT PRIMARY KEY IDENTITY(1,1),
    user_id INT,
    action VARCHAR(100),
    table_name VARCHAR(100),
    record_id INT,
    old_values NVARCHAR(MAX),
    new_values NVARCHAR(MAX),
    ip_address VARCHAR(45),
    user_agent VARCHAR(255),
    created_at DATETIME DEFAULT GETDATE()
);
```

### 4.5 Transport Layer Security

**Enkripsi SSL/TLS (WAJIB)**

- **Protokol:** TLS 1.2 minimum (TLS 1.3 preferred)
- **Sertifikat:** Sertifikat SSL valid dari trusted CA
- **Enforcement HTTPS:** Seluruh traffic redirect ke HTTPS
- **HSTS Header:** HTTP Strict Transport Security enabled
- **Perfect Forward Secrecy:** Ephemeral key exchange

### 4.6 QR Code Digital Signature

**Rujukan:** KAK Lingkup Pekerjaan - "QR Code Identity Digital Signature"

- **Tujuan:** Verifikasi sertifikat tamper-proof
- **Teknologi:** QR Code + tanda tangan digital BSrE
- **Implementasi:**
    1. Generate unique certificate ID
    2. Buat QR code dengan embedded signature
    3. Tanda tangani via BSrE API
    4. Embed dalam PDF sertifikat
    5. URL verifikasi publik

**Alur Verifikasi:**

```
Pengguna scan QR → Redirect ke halaman verifikasi → 
Verifikasi signature via BSrE → Tampilkan otentisitas sertifikat
```

```yaml
  only:
```

## 8. DISASTER RECOVERY & BUSINESS CONTINUITY

### 8.1 Strategi Backup

**Frekuensi:**

- **Database:** Full backup harian, incremental per jam
- **File Aplikasi:** Backup harian
- **Upload Pengguna:** Sinkronisasi real-time ke backup storage
- **Konfigurasi:** Version-controlled di GitLab

**Kebijakan Retensi:**

- **Backup Harian:** 30 hari
- **Backup Mingguan:** 3 bulan
- **Backup Bulanan:** 1 tahun

### 8.2 Objective Recovery

- **RTO (Recovery Time Objective):** 4 jam
- **RPO (Recovery Point Objective):** 1 jam
- **Toleransi Data Loss:** Maksimum 1 jam transaksi
